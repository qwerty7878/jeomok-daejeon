# 테스트 계획

## 레벨

| 레벨 | 도구 | 대상 |
|---|---|---|
| 단위 | Vitest | 채점 로직 (동점, 미제출, 공동우승) |
| 통합 | Vitest + Supabase local | Route Handler + DB (조건부 UPDATE 원자성) |
| E2E | Playwright **멀티 컨텍스트** | 실제 3~6명 동시 플레이 |

## ★ 반드시 있어야 할 테스트

### 원자성 (가장 중요)
```
tick 을 10개 병렬 호출 → 전이는 정확히 1회
```
이게 깨지면 라운드가 두 번 넘어간다. 아키텍처의 근간.

### 정보 누출
```
WRITING 중 GET /state  → submissions 키 자체가 없어야 함
VOTING 중 GET /state   → submissions[].author 가 undefined 여야 함
```

### 채점
| 케이스 | 기대 |
|---|---|
| 동점 최하위 3명 | 3명 전부 -1 |
| 미제출자 1명 | 자동 최하위, 투표 대상 제외 |
| 전원 미제출 | 라운드 무효, 차감 없음 |
| 마지막 2명 동시 탈락 | 공동 우승 |
| 관전자 투표 | 집계에 포함 |

### E2E (Playwright multi-context)
```
6개 브라우저 컨텍스트 = 6명
→ 방 생성 → 전원 입장 → 게임 시작
→ 4명만 제출 (2명 미제출)
→ 전원 투표
→ 미제출 2명 목숨 -1 확인
→ 1명 새로고침 → 상태 복원 확인
→ 방장 강제 종료 → 승계 확인
```
